# 会议 AI 助理(meeting-copilot)— Design v1.0

> Original design doc. Project-specific business references neutralized for public release.

**日期**:2026-05-26
**作者**:the original author + Claude(brainstorming)
**状态**:Design approved, 待 writing-plans 出实施计划
**代号**:meeting-copilot

---

## 0. 一句话

会议进行中的本地实时 AI 助理:同时抓系统音频(对方,任何 app)+ 麦克风(你自己)→ 实时转写 → 结合会前提供的资料 → 浮窗给出**话术 / 谈判分析 / 资料引用**的智能混合建议 → 会议结束生成结构化纪要。

---

## 1. 目标用户 / 场景

**主用户**:the original author(professional consulting work)。

**典型场景**:
- 客户报价谈判(Project X / 客户 A 供应商入库 / 客户 B 增项)
- 项目设计评审会(对接建筑师 / 设计院 / 业主)
- 投标方案沟通 / 立项推进
- 偶尔英文会议(海外客户 / 外籍建筑师等)

**核心痛点**:
- 现有会议工具是会后才看,不能实时辅助
- 现有工具不知道你的报价底线 / 项目档案 / 历史踩坑 / 客户性格 — 没接入私域知识
- 没有"下一步该说什么 / 该警惕什么"的决策类提示

**MVP 目标**:**单机 Mac 跑通完整流程**(从会前丢资料到会后出纪要),作者自用 5-10 次真实会议后再决定迭代方向。**不做团队多人协作**。

**分发**:每个人本地装,自己用自己的 API key。不做后端服务、不做 SaaS。

## 2. MVP 范围

### ✅ 包含

- macOS 13+ 原生(ScreenCaptureKit 免装 BlackHole)
- 同时抓「系统音频」(任何 app 输出,Zoom/腾讯会议/线下用麦)+「麦克风」
- 阿里 Paraformer-realtime-v2 实时转写(中英混说原生,行业热词预注入)
- **会议前临时丢文件做 RAG**(scope A)— 拖入 PDF/Word/Markdown,会议结束即丢
- **副驾驶浮窗**(模式 A)— always-on-top,半透明,180×360 px
- **智能混合建议**(风格 4)— LLM 动态切换战术 / 战略 / 信息 三种风格
- 每 15-30s 自动建议 + 快捷键召唤详细建议
- 会议纪要 GPT-4o 生成(结构化:摘要 / 决议 / Action Items / 风险 / 引用)
- 历史会议浏览(列表 + 详情 + 重新生成纪要)
- 导出:Markdown / Obsidian / 飞书文档(MVP 仅 Markdown,后两个 Phase 1.5)
- macOS Keychain 存 API key

### ❌ 不含(Phase 2 backlog)

- Windows 端(架构已抽象 OS-specific 音频层,加 WASAPI 模块即可)
- Speaker diarization(谁说了哪句话)
- 自动写入 Obsidian / 飞书 base
- 智能预警(浮窗自动展开 + 提示音)
- iPad / iPhone 副屏
- 多人协作 / 团队会议室
- 长期资料库(全量接入 Obsidian / 飞书 / 规范库)
- 多语言 UI(MVP 中文)
- 离线模式(本地 Whisper)

## 3. 用户旅程

### 3.1 会议前(Setup)

```
打开 app
  → 新建会议
  → 输入:会议名(必填)/ 关联项目(可选, 选了则自动加载关联档案)
       / 会议目的(可选, 影响建议风格)/ 参会人(可选)
  → 拖入资料文件(PDF / Word / MD / TXT, 多文件 OK)
  → 后台:文件 → 文本提取 → 切块 → embedding → SQLite 入库
  → 进度条显示"资料索引中... 3/5 完成"
  → 全部完成 → "开始会议"按钮可点
  → 点击 → 倒计时 3s → 浮窗出现 + 录音/转写启动
```

### 3.2 会议中(Live)

```
浮窗显示在右下角:
  - 顶部:状态灯(ASR 正常/重连)+ 静音 + 结束按钮
  - 中上部:实时滚动转写(5-6 行小字, 区分"对方"/"我")
  - 中下部:当前建议卡片(动态切风格)
  - 底部:引用资料标签(点开 → 弹窗显示原文片段)

每 15-30s:
  - LLM 静默判断"现在该不该出建议"(避免话题刚起就建议)
  - 该出 → 拼 prompt → MiniMax stream → 卡片更新
  - 不该 → 跳过, 等下个窗口

用户按 Cmd+Shift+M:
  - 立即触发一次"详细建议"(用更强 prompt + 更多 context)
  - 卡片展开成大模态(可见 90s 内的转写 + 建议 + 资料)
```

### 3.3 会议结束

```
按 Cmd+Shift+R 或 浮窗"结束"按钮
  → 停 ASR / 录音
  → 提示"生成纪要中..."(等 30-60s)
  → 调 GPT-4o, 输入全量转写 + 会议元数据 + 资料引用
  → 产出结构化 Markdown 纪要
  → 自动存 SQLite + 显示在历史列表
  → 弹"纪要已生成", 提供按钮:查看 / 复制 MD / 导出文件 / 导出到 Obsidian (Phase 1.5)
```

## 4. 架构

### 4.1 三进程隔离

```
┌─────────────────────────────────────┐
│  Frontend (React + TypeScript)      │
│  - Setup 页 / 浮窗 / 历史会议页       │
│  - 跑在 Tauri webview                │
└──────────────┬──────────────────────┘
               │ Tauri IPC (invoke / emit)
┌──────────────▼──────────────────────┐
│  Tauri Main (Rust)                  │
│  - 业务逻辑:ASR/RAG/LLM 编排         │
│  - SQLite (会议/纪要/资料/embedding)  │
│  - macOS Keychain 读取               │
└──────────────┬──────────────────────┘
               │ stdio pipe (PCM 16kHz frames + JSON control)
┌──────────────▼──────────────────────┐
│  AudioHelper (Swift CLI binary)     │
│  - ScreenCaptureKit(系统音频)        │
│  - AVAudioEngine(麦克风)             │
│  - 混流 + 重采样到 16kHz mono PCM    │
└─────────────────────────────────────┘
```

**为什么 Swift 独立子进程而非 FFI dylib**:
- Swift ↔ Rust FFI 桥接前期投入 3-5 天,且调试链路长
- 独立 binary 崩溃不连累主进程,权限边界清(屏幕录制权限只授给 helper)
- PCM 16kHz mono 带宽 ~32KB/s,IPC 开销可忽略
- MVP 选稳,后期想换 FFI 不破坏其他层

### 4.2 数据流(会议中)

```
[系统音频 + 麦克风]
    │
    ▼
[Swift AudioHelper]
    │ stdout: PCM 16kHz mono frames + 'src' tag (system/mic)
    ▼
[Tauri Rust: AudioPump]
    │ 转发 + 备份(同步写本地 .wav 录音)
    ▼
[阿里 Paraformer WebSocket]
    │ 流式回复 transcript chunks (含 speaker label / 时间戳)
    ▼
[Rust: TranscriptBuffer]
    │ 累积 + 推送前端浮窗(实时滚动)
    │
    │ 每 15-30s 定时器触发:
    ▼
[Rust: SuggestionEngine]
    │ 1. 取最近 90s transcript
    │ 2. RAG 检索 top-5 相关资料块(基于最近 30s 关键句的 embedding)
    │ 3. 拼 prompt(系统提示 + 会议元数据 + transcript + 资料 + 智能混合指令)
    │ 4. 调 MiniMax abab6.5 stream
    │ 5. SSE 推到前端浮窗(token-by-token 显示)
    ▼
[Frontend: 浮窗]
    动态渲染建议卡片
```

### 4.3 数据流(会议结束)

```
[结束触发]
    ▼
[停 ASR / 停 AudioHelper / 关闭 WAV 文件]
    ▼
[Rust: MinutesGenerator]
    1. 全量 transcript 拼装
    2. 会议元数据(名/项目/参会人/目的) + 当时建议历史摘要
    3. 调 GPT-4o (gpt-4o, ~50K tokens 上下文)
    4. 系统 prompt 要求结构化输出:
       - ## 摘要 (3-5 句)
       - ## 关键决议 (bulleted)
       - ## Action Items (Markdown checklist, 含 @谁 / 截止)
       - ## 风险与红线
       - ## 引用资料
       - ## 完整转写(折叠)
    ▼
[SQLite: 写入 minutes 表]
    ▼
[Frontend: 历史会议页刷新]
```

## 5. 组件详解

### 5.1 AudioHelper(Swift CLI)

**职责**:抓系统音频 + 麦克风,混流输出 16kHz mono PCM。

**接口**:
- stdin:JSON 控制命令(`{cmd: "start"}` / `{cmd: "stop"}` / `{cmd: "mute_mic"}`)
- stdout:二进制 PCM 帧(每帧前 8 字节 header:`[u32 source_tag][u32 frame_size]`,source_tag = 0 系统 / 1 麦克风 / 2 混流)
- stderr:JSON 日志

**关键实现**:
- `SCStream`(ScreenCaptureKit) 抓系统音频,`SCContentFilter` 排除当前 app 自己的输出
- `AVAudioEngine.inputNode` 抓麦克风
- `AVAudioConverter` 重采样到 16kHz mono Int16
- 双源**可选混流**或**双轨输出**:MVP 输出双轨,Rust 端可分别送 ASR(让 ASR 同时识别"对方说"和"我说"两条流,准度更高)

**权限**:
- Info.plist 声明 `NSScreenCaptureDescription` + `NSMicrophoneUsageDescription`
- 首次使用触发系统弹窗,被拒后引导用户去「系统设置 → 隐私与安全 → 屏幕录制 / 麦克风」

**编译产物**:`AudioHelper` binary,放 Tauri app bundle 的 `Contents/Resources/` 下,Rust 启动时绝对路径调用。

### 5.2 Tauri Main(Rust)

**核心模块**:

| 模块 | 职责 |
|---|---|
| `audio_pump` | 启动 AudioHelper 子进程,读 stdout,转发到 ASR + 写本地 WAV 录音 |
| `asr_client` | 阿里 Paraformer WebSocket,断线重连,热词表注入,trait 抽象后期可换 |
| `transcript_buffer` | 滚动 transcript 状态(in-memory + 定期 SQLite 持久化) |
| `rag_engine` | 文件解析 / 切块 / embedding / 检索;sqlite-vec 后端 |
| `suggestion_engine` | 定时器 + 召唤,拼 prompt,调 MiniMax,推前端 |
| `minutes_generator` | 会后调 GPT,生成结构化 Markdown,写 SQLite |
| `db` | SQLite schema 管理 + migrations |
| `keychain` | 读写 macOS Keychain(API key) |

**LLM 抽象**:统一接口 `LLMClient { generate(model, system, messages, stream) }`,实现层:`MiniMaxClient` / `GPTClient` / 留 `ClaudeClient` 占位。换厂商改一行配置。

**ASR 抽象**:`ASRClient { connect, push_pcm, on_transcript }`,实现 `AliyunParaformer` / 留 `Xfyun` 占位。

### 5.3 Frontend(React + TypeScript + Tailwind)

**三个 window**:
- **Setup window**(Tauri main window):800×600,会议前配置
- **Floating window**(Tauri secondary window):180×360,会议中浮窗,`alwaysOnTop: true`, `decorations: false`, `transparent: true`
- **History window**(可通过菜单栏 → 历史 触发):900×700,会议列表 + 详情

**状态管理**:轻量,Zustand 或 React Context 就够,不需要 Redux。

**实时通信**:Tauri 的 `event` API(Rust `emit` → JS `listen`)流式推 transcript / 建议 / 状态。

### 5.4 RAG Engine

**文件解析**:
- PDF:`pdf-extract` crate(纯 Rust,够用)
- Word:`docx-rs` crate
- Markdown / TXT:直接读
- 加密 PDF / 图片 PDF(扫描件):MVP 暂不支持,提示用户先 OCR

**切块策略**:500 字符 / 50 字符 overlap,按句子边界对齐(中英文 punctuation 都识别)。

**Embedding**:**阿里 text-embedding-v3**(1024 维,跟 ASR 同栈)。
- 批量入库时调 batch API
- 检索时实时 embed 查询句

**存储**:SQLite + `sqlite-vec` extension。Schema:
```sql
CREATE TABLE chunks (
  id INTEGER PRIMARY KEY,
  meeting_id TEXT NOT NULL,
  source_file TEXT NOT NULL,
  chunk_index INTEGER NOT NULL,
  text TEXT NOT NULL,
  created_at INTEGER NOT NULL
);

CREATE VIRTUAL TABLE chunks_vec USING vec0(
  embedding float[1024]
);
```

**检索**:cosine top-5,K 可调。检索 query = 最近 30s transcript 用 LLM 抽出的"关键句"(避免拿整段 90s 检索噪音大)。

## 8. 数据模型(SQLite)

```sql
-- 会议
CREATE TABLE meetings (
  id TEXT PRIMARY KEY,              -- UUID
  name TEXT NOT NULL,
  project_ref TEXT,                 -- 关联项目名(自由文本)
  purpose TEXT,                     -- 报价谈判 / 设计评审 / ...
  participants TEXT,                -- JSON array
  started_at INTEGER NOT NULL,      -- Unix ms
  ended_at INTEGER,
  audio_path TEXT,                  -- 本地 WAV 路径
  metadata TEXT                     -- JSON, 扩展字段
);

-- 资料文件(会议级)
CREATE TABLE materials (
  id TEXT PRIMARY KEY,
  meeting_id TEXT NOT NULL REFERENCES meetings(id),
  file_name TEXT NOT NULL,
  file_path TEXT NOT NULL,
  file_size INTEGER,
  indexed_at INTEGER,
  chunk_count INTEGER
);

-- 切块(RAG)
CREATE TABLE chunks (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  meeting_id TEXT NOT NULL,
  material_id TEXT NOT NULL,
  chunk_index INTEGER NOT NULL,
  text TEXT NOT NULL
);

CREATE VIRTUAL TABLE chunks_vec USING vec0(embedding float[1024]);

-- 转写记录
CREATE TABLE transcripts (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  meeting_id TEXT NOT NULL,
  speaker TEXT,                     -- 'system'(对方) / 'mic'(我) / 'unknown'
  text TEXT NOT NULL,
  start_ms INTEGER NOT NULL,        -- 相对会议开始
  end_ms INTEGER NOT NULL
);

-- 建议历史(会议中给过的建议)
CREATE TABLE suggestions (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  meeting_id TEXT NOT NULL,
  triggered_at INTEGER NOT NULL,
  trigger_type TEXT,                -- 'auto' / 'manual'
  style TEXT,                       -- 'tactic' / 'strategy' / 'info' / 'mixed'
  content TEXT NOT NULL,            -- JSON, 含 message + cited chunks
  user_action TEXT                  -- 'viewed' / 'ignored' / NULL
);

-- 纪要
CREATE TABLE minutes (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  meeting_id TEXT NOT NULL,
  version INTEGER NOT NULL,         -- 重新生成自增
  markdown TEXT NOT NULL,
  generated_at INTEGER NOT NULL,
  model_used TEXT,                  -- 'gpt-4o' / ...
  tokens_used INTEGER
);
```

## 9. API 集成详情

### 9.1 阿里 Paraformer-realtime-v2

- **端点**:`wss://dashscope.aliyuncs.com/api-ws/v1/inference/`(WebSocket)
- **协议**:JSON 控制帧 + 二进制音频帧
- **音频格式**:16kHz mono PCM 16-bit
- **热词注入**:每次连接时 send `task-start` 消息携带 `vocabulary_id`(预先用 phrase manager API 上传行业术语表得到)
- **行业术语表来源**(示例,按使用者所在行业自行准备):
  - 项目代号、缩写
  - 主要客户 / 合作方姓名
  - 建筑师事务所 / 设计院 / 总包 等机构名
  - 行业专业术语
- **断线重连**:指数退避(1s / 2s / 4s),3 次后提示用户
- **文档**:https://help.aliyun.com/zh/model-studio/websocket-for-paraformer-real-time-service

### 9.2 MiniMax 实时建议

- **端点**:`https://api.minimax.chat/v1/text/chatcompletion_v2`(国内)
- **模型**:`abab6.5-chat`(平衡)或 `abab6.5s-chat`(更快更便宜)
- **stream**:SSE
- **typical prompt 结构**:
  ```
  系统:你是用户的会议助理...(完整角色 + 风格混合策略)
  用户:
  ## 会议元数据
  {meta}
  ## 最近 90s 转写
  {transcript}
  ## 相关资料(top-5)
  {chunks}
  ## 任务
  根据当下对话节奏,从 [战术 / 战略 / 信息] 中智能选最该给的风格,产出 1 条建议(<200 字),含话术或分析或数据,引用资料 1-2 条。
  ```

### 9.3 GPT 纪要

- **模型**:`gpt-4o`
- **预估 tokens**:1 小时会议 ~ 15000 字 ≈ 25K tokens 输入,5K tokens 输出,单次成本约 ¥1-2
- **prompt 结构**:见 §5.2 `minutes_generator`

## 10. 权限 / 错误处理

| 场景 | 处理 |
|---|---|
| 首次启动 | 引导页申请屏幕录制 + 麦克风权限,一键跳转系统设置 |
| 屏幕录制被拒 | 顶栏红色横幅,无法开始会议;点击 → 系统设置 |
| ASR WebSocket 断 | 自动重连(指数退避 3 次);失败 → UI 黄色横幅 "ASR 暂停, 录音继续";恢复后批量补转写 |
| LLM API 失败 | 这一轮建议跳过;3 轮连续失败 → 黄色横幅 "LLM 暂时不可用" |
| 网络全断 | 录音继续,ASR 暂停;网络恢复后用阿里 Paraformer 文件转写 API 批量补转写;Phase 2 加本地 Whisper 离线兜底 |
| 磁盘满 | 录音停止,提示用户清理 |
| 资料文件损坏 | 单文件失败不影响其他,标红 + skip |

## 11. 隐私 / 安全

| 数据 | 流向 |
|---|---|
| 系统音频 + 麦克风 PCM | 本地内存 → 阿里 Paraformer(境内, 有 DPA, 实时流不存储)→ 本地 WAV 备份 |
| Transcript 文字 | 本地 SQLite + MiniMax(国内, 实时建议用)+ GPT(境外, 仅会议结束纪要生成用) |
| 资料文件文本 | 本地 → 阿里 embedding API(单向, 出向量)→ 本地存向量,资料原文不出境 |
| 资料原文 chunks | 检索时进 MiniMax / GPT prompt(注意:这条边界 — 会议纪要传 GPT 时, 引用的 chunks 文字会出境) |
| API keys | macOS Keychain |
| 录音 WAV | 本地 `~/Library/Application Support/meeting-copilot/recordings/` |

**默认保留策略**:
- 录音 WAV:30 天后自动删
- Transcript / Minutes:永久(SQLite)
- 资料 chunks:会议结束后保留 30 天(支持回看时引用),超期清

**用户告知**:首次启动弹一次"隐私说明",列清楚以上数据流。后续在「设置 → 隐私」可查看 + 调保留期。

## 12. 测试策略

**单元测试**:
- RAG 切块边界 / embedding 长度 / 检索准确性(用 fixture chunks)
- ASR client 重连逻辑(mock WebSocket)
- LLM client 接口(mock API)
- Prompt 拼装(snapshot test)

**集成测试**:
- 用预录 fixture 音频(中文 30s + 中英混 30s + 英文 30s)喂完整 pipeline,验证 transcript → suggestion 路径
- 录音回放 → 纪要生成,验证输出结构

**真实测试(MVP 验收)**:
- 作者拿 **3-5 场真实会议**跑 MVP
- 验收点:
  1. ASR 识别准确率 ≥ 90%(尤其行业术语 / 客户名 / 项目编号)
  2. 至少 50% 的建议是「我看完会用 / 受启发」
  3. 浮窗在会议中**不让你分神 / 不被对方注意**
  4. 纪要质量达到「我可以直接发给同事 / 改 5 分钟」
  5. 端到端延迟 ≤ 15s(对方说完到看到建议)

## 13. 我替你拍的小决策(可推翻)

1. **录音保留 30 天** — 本地,可设
2. **MVP 不做 speaker diarization** — 阿里自带的 speaker label 用于"对方/我"区分够了,精细化等 Phase 2

6. **MVP 录音 WAV 本地存** — 不加密。Phase 2 加密可选
